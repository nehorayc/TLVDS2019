Intro to Git & Unix utilities

In this session we will explain how to use the most common Unix utilities, including git, our friendly Version Control System (VCS).

To start , we will clone a program repository, this means making a copy of a repository in github in our computer.

Clone a repository from Github with git clone:

git clone https://github.com/nehorayc/TLVDS2019.git

This will download the repository into a folder (also named directory) named ixperience_2019_instructor.

You can check that the folder exist by using ls , that lists the files in the current folder:

ls

Now we will create another directory, and this one will be our working directory. We can do that with the command mkdir

mkdir ixperience_2019_personal

We will now navigate into the newly created directory with cd (change directory):

cd ixperience_2019_personal

We can check the directory we are by doing pwd.

pwd

Working with your own repository

Now that we are in a new directory we can create a new git repository, this will allow us to keep files in this directory in version control (securely stored in Github).

Start a repository with git init.

You also need to create it in Github by clicking on the New Button:

Make sure you name your repository ixperience_2019_personal

Add a remote to the new repository with git remote add origin REMOTE_URL.

Where origin denotes the name you will use to identify this remote in this repository (you can have multiple remotes, for example, one can be your personal copy of a project and another can be your company's copy of a project).

Where REMOTE_URL will be your repository's github url (usually https://github.com/YOUR_GITHUB_USERNAME/ixperience_2019_personal.git)

We are going to make another directory to store today's session files and navigate into it. We can execute multiple commands consecutively by using && to separate them. For example, if we want to create a directory and move into it we can do:

mkdir intro_terminal && cd intro_terminal

Use mv to move files from one folder to another, and cp to copy files from one folder to another

For example, we can copy the file primary_results.csv from my repository into your personal folder.

cp ../../ixperience_2019_instructor/intro_terminal/primary_results.csv .

Note: For the previous path, we are using some unix path conventions, where .. means "the directory above the current one" and . means "the current directory".

So what we are actually doing is:

COPY "one folder above current one -> another folder above the folder above current one -> folder ixperience_2019_instructor -> file primary_results.csv"  INTO the current folder

Add files to a new commit (commit them) with git add FILE_NAME

git add primary_results.csv

Thats it! We have used git to clone someone else's repository, we have copied a file from that repository into our brand hew repository, and we have added that file to a new commit!.

Now we can "commit" the file so its tracked on our local git repository:

git commit -m 'added primary results file'

NOTE:We can use rm to remove files (we need to use the flags -r to remove folders recursively:

mkdir wrong_folder
rm -r wrong_folder

Unix utilities Overview.

Where we see the most common Unix commands and how to use them.

ls (list)

we use ls to list items in a folder

ls

We can add arguments to commands, that provide different functionalities (or enable different usage modes). For example, if we want to list the files in a folder and see their file sizes we can do:

ls -lh

where the argument -lh is adding two options together l which provides aditional information for each file and h that provides the file sizes in human readable format (megabytes instead of bytes)

man (manual)

Most unix applications have a user manual that you can check for instructions on how to use it. For example, we can do:

man ls

touch

we use touch to create a new file. For example, we can create a README.md file (a file that provides a description for a repository) like:

touch README.md

By the way, now that we have created the readme file, lets open it with any text editor and write a repository description! (for example "ixperience Program Repository"), then save it!.

wc (word count)

We use wc to count elements in a document (or documents) We can use wc with the argument -l to just count the number of lines in a document

wc -l primary_results.csv

We can count the lines for all files in the folder if we use a wildcard (*) like:

wc -l *

head

We use head to read the first lines in a file, it is useful to get a feeling of how a file looks like

head primary_results.csv

We can see a specific number of lines if we use the option -n, for example, if we want to see the first 5 lines.

head primary_results.csv -n 5

tail

We use tail similarly to head, but to print the last lines in a file

tail primary_results.csv -n 5

cut

We use cut to split lines in a file based on a delimiter ( a delimiter being a character that delimits the different fields)

For example, if we want to split every line of the file by its commas, and get the second field, we can do so like:

cut -d "," -f 2 primary_results.csv

grep

grep allows us to search specific strings (or regular expressions) in a file.

grep "New York" primary_results.csv

We can grep lower and upper case words with the argument -i

grep -i "new york" primary_results.csv

sort

We can use the command sort to well, sort files.

We can sort by a specific column with the argument -k and specify the field separator with -t

So if we want to sort by the 3rd column, we can do it like:

sort -k 3 -t "," primary_results.csv

cat

cat prints the contents of a file to the terminal

pipes (>, >>, and |)

The cool thing about most unix utilities is that they are composeable, that means that we can use the output of one command as the input to another! This works because all of unix utilities print their output in what is called Standard Output (stdout), one of the three basic standard streams, the other two being standard input and standard error :

We compose utilities by "piping" the commands using the pipe |

For example we can use cut to find the 2nd column of the file then get the first 5 elements using head

cut -d "," -f 2 primary_results.csv | head -n 5

We can use the greater than > symbol to write the current output into a file

For example, if we want to save the output of the previous command into a file we can do that like:

cut -d "," -f 2 primary_results.csv | head -n 5 > first_5_states.csv

We can now check the file:

cat first_5_states.csv

Using > will recreate the file from scratch, while >> will append to the file (add lines to the end)

uniq

We can use uniq to find the uniquely different lines in a list of lines (and count their ocurrences with -c)

cut -d "," -f 2 primary_results.csv  | uniq -c

However, uniq is not a smart command. Repeated lines will not be detected if they are not adjacent. So you have to sort the file before hand.

Making a commit

We will add a couple of additional files into our commit:

git add README.md && git add first_5_states.csv

We can check the status of our repository with git status.

git status

If things went well, we should see that 3 files are going to be added to the repo.

Once we have added all the files we want to track into version control, we need to commit them. Commit basically means telling git "ok, this is the new version for the files that I want you to track".

Make a commit with git commit -m 'COMMIT_MESSAGE'

git commit -m 'Add intro to terminal example files'

Push the changes (the new commit) to Github with git push REMOTE_NAME BRANCH_NAME

git push origin master

Where origin is the name we assigned to our remote when we added it and master is the default branch.

Python from the terminal

There are 2 main ways to interact with python from the terminal.

REPL Mode

First, you can run python's REPL (Read-Eval-Print-Loop) , also called Python Console, Python Terminal. This opens a console where you can interact similarly to the CodeAcademy's tutorial. You open the repl just typing python.

You can exit the repl by pressing (ctrl+d) or (ctrl+z), it depends on the OS.

Script Mode

We can also call python to run a python script. What is a python script? Basically, its a text file, with the file extension .py, that contains python code that we can run.

We can run those scripts just by doing

python name_of_the_script.py

By doing that we will run ALL OF THE CODE IN THE SCRIPT

For example, lets create a python script that prints out the current date. We can create the script by doing:

touch current_date.py

Inside this script we put the following:

from datetime import datetime
print(datetime.utcnow())

If we run this script (by doing python current_date.py we see that it prints the current date (in UTC timezone).

And because the output for python is written to standard output, we can pipe it too!

 python current_date.py | cut -d "-" -f 1

SSH

One of the key differences between statisticians and data scientists is that statisticians usually work in a closed environment (where data is feed up to them and they are not in charge of the infrastructure).

Data Scientists (specially the first ones to join a company) usually have to set up infrastructure. In practice this means connecting to machines to set them up and to run some code.

Connections to machines are usually done by ssh (Secure Shell). SSH allows us to connect to a machine and use shell commands there. We usually say "im going to ssh to machine X"

In order to ssh to a machine, we just need to do:

ssh user@machine

Usually if we ssh like that the machine will ask for our password and will let us in.

I set up a toy machine (on the cloud, waaaat!!?) where you can ssh.

To do so, just ssh using the password, student:

ssh student@35.180.105.151

Other times we will need to provide a key to authenticate ourselves (this is called public key authentication). Public key authentication is a more robust way of authentication than using a password. Every key (keypair), consists of 2 pieces, a public key (that can be shared publicly) and a private key (that must be safely stored). Owners of a private key can create messages (encrypt) using that key and readers of the public key can validate that the private key is valid, but not the other way around. This way a user can ssh to a machine without having to remember a password.

You can follow the steps on this github tutorial to create your ssh keypair.
